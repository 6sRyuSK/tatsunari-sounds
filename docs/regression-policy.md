# 退行防止規定 — Issue #22〜#44 から得た恒久ルール

このドキュメントは、Issue #22〜#44 で顕在化したバグ群を **クラスとして** 再発
させないための規定です。個々の修正はすでに完了しています。ここで定めるのは
「同じ **種類** のバグを、次に作るプラグイン / core 変更で二度と通さない」ための
不変量とゲートです。

- 対象読者: この工場で新しいプラグインを追加する / `core/` を変更する人（および
  エージェント）。
- 位置づけ: `CLAUDE.md` のハードルールを補足する詳細版。矛盾する場合は
  `CLAUDE.md` が優先。
- 共有ヘルパ: 数値安全・サンプルレート・解像度の不変量は
  `core/include/factory_core/testing/DspInvariants.h` に集約済み。新規／変更テストは
  これを使うこと（手書きでレート集合を狭めない）。

> **ゲートを緩めて緑にするのは禁止（`CLAUDE.md`「Ask a human」）。** 本規定は
> ゲートを **足す** ものであり、既存の tolerance / oracle / disabled-tests の
> 緩和はすべて人間承認が必要。

## クラス別・具体ルール

### A. フィードバック発散させない（#28, #29）
- フィードバックを持つ DSP は、**あらゆる正規レンジ設定でループゲイン < 1** を
  設計上保証すること。COLA 正規化のように「特定点でちょうど 1」になる式は、
  その点を外れた領域（疎オーバーラップ等）で 1 を超えないよう **min クランプ**
  を入れる（例: `gain = min(1.0, 2*interval/dur)`）。
- freeze / hold のような損失ゼロ経路には **外部エネルギーを注入し続けない**
  （freeze 中は入力も帰還も断つ）。
- **テスト（必須）**: `impulseResponseNonIncreasing()` を使い、**最悪ケース設定**
  （最大 feedback・疎オーバーラップ・最長 decay 等）でインパルス応答エネルギーが
  非増加であることを **全6レート** で検証する。50%オーバーラップ等の「たまたま
  安定な一点」だけを試験しない（#34）。

### B. 共振ピークに絶対上限（#36）
- 高次フィルタカスケード（HP/LP スロープ×Q）は、ピーク段の Q / 共振倍率に
  **段数の関数で上限** を設けること（例: `min(q*userQ/butterQ, 8.0)`）。Butterworth
  応答は不変に保つ。
- **テスト（必須）**: 最大スロープ×最大 Q で、カットオフ近傍のピークゲインが
  **既知の絶対上限内**（例 ≤ +18dB）であることを assert。「ピーク>0dB」だけでは
  上限を縛れない。z-domain `H(e^jω)` で評価（アナログ prototype 不可）。

### C. finite ガードで自己回復（#38）
- すべての帰還ノード（FDN・DelayLine・PitchShifter の帰還）に軽量な
  `std::isfinite` ガードを置き、NaN/Inf を検出したらその経路をゼロ/フラッシュ
  して **自己回復** させること。`reset()` 頼みにしない。
- **テスト（必須）**: 高帰還・長 decay で数十秒ホールドし、`allFinite()` が保たれ、
  かつ peak が現実的な上限内（`peak < 10.0` 等、`1e6` のような「NaN でないだけ」の
  許容は不可、#35）であることを検証。

### D. バッファは最悪ケースで確保（#37）
- 遅延/リングバッファ長は、**到達し得る最大遅延**（テンポシンク最小 BPM×最大拍
  ＋ LFO/変調ヘッドルーム）から導出すること。無音クランプで誤動作させない。
- **テスト**: 最悪ケース（遅い BPM 等）で実効遅延が期待どおり（クランプされて
  いない）ことを検証。

### E. 状態リセット漏れをなくす（#30, #39, #41）
- `prepareToPlay` の末尾で DSP の `reset()` を呼ぶ（再 prepare / サンプルレート
  変更で古い z-state を残さない）。core クラスには `reset()` を用意し、
  複合クラス（`MultibandCompressor`→`Crossover3` 等）は子まで伝播させる。
- **bypass 解除** と **channel/mode 切替** の遷移を検出し、idle だった経路を
  `reset()`（またはドライへクロスフェード）してからアクティブ化する。
- レイテンシ申告プラグインは bypass 中も整合を保つ（未遅延ドライを申告レイテンシ
  のまま出さない、#30）。

### F. 連続パラメータは平滑化（#31, #32, #40）
- 状態を持つ係数/ゲイン/カットオフを **毎ブロック不連続に差し替えない**。
  `juce::SmoothedValue`（対数域が自然ならログで）でサブブロック（例32サンプル）
  粒度に平滑化、またはゲインは出力をランプ、係数は current→target をクロスフェード。
- 平滑化の状態・バッファは **`prepareToPlay` で確保**（`processBlock` で確保・
  ロック・syscall しない = RT安全ハードルール）。

### G. サンプルレート＆解像度（#33, #16）
- 全プラグインの DSP テストは **44.1 / 48 / 88.2 / 96 / 176.4 / 192 kHz** の
  全6レートで走る。CMake の `add_test` ループとテストの既定レートは
  `factory_core::testing::kStandardSampleRates()` /
  `sampleRatesFromArgs()` に一致させる（手書きで狭めない）。
- FFT/STFT/アナライザーのサイズは **サンプルレートから導出**
  （`fftOrderForSampleRate`）。固定オーダー禁止。
- **テスト（必須）**: `resolutionFollowsSampleRate()` で、最上位レートでも
  ビン幅（fs/N）と窓長（N/fs）が範囲内であることを assert。

### H. テストは必ずバグ経路を踏む（#34, #35, #42, #43）
- **バグを実際に踏むケース** を書く。境界の一点（COLA/50%オーバーラップ、
  ゼロ shimmer）だけを試験しない。
- サンプルレートループの本体は **`Fs` に実際に依存** させる（信号の周波数/長さ・
  帯域/エイリアス上限を Fs から導出）。no-op ループ禁止（#42）。
- 量的な検証には **実装から独立した静的オラクル** を持つ（例 コンプの
  `GR=(L-T)(1-1/R)`）。「1%以上減衰した」のような有無判定だけにしない（#43）。
- 安定性許容は現実的な上限に（`1e6` のような値は実質「NaN でないだけ」）。

### I. スレッド間共有は atomic（#44）
- audio/prepare スレッドが書き、GUI が読む共有スカラは `std::atomic`
  （表示用途なら `memory_order_relaxed` で可）。非 atomic の平文共有を残さない。

### J. 検出器に絶対 floor（#24）
- スペクトル検出・ゲート等は、相対超過だけでなく **絶対レベル floor**
  （例 -80dBFS）を持ち、ノイズフロア以下の入力には反応しないこと。
- **テスト（必須）**: 無音/極小レベル入力で **リダクション/検出が出ない**
  （silence テスト）ことを全レートで検証。

### K. GUI は共有ヘルパで整合（#22, #23, #25, #26, #27）
- 数値表示の小数桁は共有ヘルパ（`factory_ui::setSliderDecimals` 等）で固定し、
  JUCE `SliderAttachment` の桁上書きを打ち消す（Freq は整数 Hz、その他は小数第2位）。
- カーブとノードは **同一の描画空間**（同じベースライン/スケール）で描く。
- トグル/ラベルは全文が収まる幅を確保する。

### L. 位相コヒーレント並列合成（コムフィルタ回避）
- ドライ（direct）とウェット（enhance/wet）を並列に混ぜるプラグインは、両経路の
  位相とレイテンシを **構造的に** 一致させること。クロスオーバーで帯域分割する
  場合、direct 経路も帯域と同一の allpass カスケード（`Crossover5::allpass()` 等）に
  通し、両者を **同一のオーバーサンプリングブラケット内** で処理する。ゲインを掛けた
  ドライを別レイテンシ/別位相で足すと、合成時にコムフィルタ（周波数依存の谷）が出る。
- **テスト（必須）**: enhance=0・width=100 で (a) direct のみ 0dB → 帯域全域で
  **平坦**（0dB±許容）、(b) direct 0dB かつ wet 0dB → **+6.02dB±許容**（コム縮みが
  あれば即死）を **全6レート** で assert。クロスオーバーを最小比に詰めた設定でも
  確認する（インパルス→FFT で位相ではなく振幅を見る）。

### M. 線形位相クロスオーバー（完全再構成・RT安全・レート連動）
- 線形位相 FIR で帯域分割する場合、**相補性を構造で保証** する: 4 本の線形位相
  ローパス（同一長 N＝同一群遅延 D）を用意し、隣接ローパスの **差** と D サンプル
  遅延した入力で 5 帯域を作る。帯域和は望遠鏡和で **純遅延（D サンプル）に厳密に
  一致** する（`LinearPhaseCrossover5`）。線形モードの direct 経路はその純遅延なので、
  帯域和と位相コヒーレント（クラス L の線形位相版）。
- **FIR タップ長は固定禁止・レート連動**（クラス G）。群遅延を一定時間（例 ~43ms）に
  保つよう N をサンプルレートから導く。申告レイテンシ = (N-1)/2 ＋ OS ラウンドトリップ。
- **FIR 再設計は `processBlock` 内でやらない**（窓関数 sinc 設計＋FFT は重く allocation
  する = RT ハードルール違反）。ドラッグ可能なクロスオーバーはメッセージスレッドで
  予備係数バンクに設計し、atomic 1 回で **lock-free に差し替える**（`process` は active
  バンクを読むだけ）。位相モード切替はレイテンシが大きく変わるので、20ms クロスフェード
  ではなく **reset ベース＋即時 PDC 再申告**（~43ms 差を跨ぐフェードはコムになる）。
- **テスト（必須・全6レート）**: (a) 帯域和==純遅延（インパルス→和→FFT が平坦、極端な
  fc 40Hz/18kHz でも）、(b) タップ対称＋回転後の虚部消失で **群遅延平坦（線形位相）**、
  (c) fc で **-6dB**（相補クロス点）＋ストップバンド、(d) 申告レイテンシ=(N-1)/2+OS かつ
  タップ数がレート連動、(e) 線形モードでも enh0 フラットネス・最悪ケース finite・
  ドラッグ再設計後の平坦を確認。既存ゲート（標準位相）を **緩めない**。

### N. バイパスは申告レイテンシを保存する（PDC・クラス E のレイテンシ版）
- レイテンシを申告する STFT/FIR プラグインは、**バイパス中も申告レイテンシを保存**
  すること。processBlock でバイパス時に早期 return して未遅延ドライを出すと、申告
  レイテンシ（N サンプル ≈ 数十 ms）分だけ他トラックとずれ、切替のたびに PDC が
  壊れる。
- 実装: バイパス中もエンジンを**回し続け**（フレーム処理・ゲイン更新・表示スナップ
  ショットは従来どおり継続）、**出力段だけ**を申告レイテンシ整列済みドライへ短い
  ランプ（例 10ms、ステップは prepare で事前計算し per-sample 除算しない = RT安全）で
  **線形クロスフェード**する。ドライ/ウェットは高相関なので等パワーにしない。完全
  バイパス時はドライを演算を通さず**ビット透過**で出す。`reset()` はランプ値を
  ターゲットへスナップし、prepare 直後にフェードを走らせない。
- **テスト（必須・全6レート）**: (a) 起動時からバイパスで全 n で `out[n]==in[n−N]` を
  **厳密等価**（最初の N はリングのゼロ）; (b) 常時 active と途中でバイパス ON→OFF する
  双子で、バイパス完了区間は出力＝整列ドライ（厳密）、解除＋ランプ後は active 双子と
  **厳密一致**、ランプ区間も両端値のポイントワイズ凸結合境界内、かつ全サンプル有限。

### O. Delta モニタは Mix と相補（Delta·Mix 恒等）
- Delta（removed-signal モニタ）を持つプラグインは、normal 出力と**相補**な
  removed 信号を出すこと。dry−wet 全量を出すと、Mix<1 のとき実際に除かれている音
  より大きい音を聴くことになり、「delta で聴いたものがそのまま Mix で消えている」
  というモニタの意味が壊れる。
- **Mix はスペクトラルゲイン領域に乗せる**（時間領域 dry/wet ブレンドにしない）:
  実効ゲイン `gEff[k] = 1 + mix·(g[k]−1)`（g = バリスティクス済み per-bin ゲイン）を
  フレーム毎に適用する。これは後続のデュアル解像度化（低解像 dry × 高解像 wet）で
  dry と wet の位相差コムを**構造的に**防ぐ前提（Mix を時間領域で混ぜると別解像度・
  別位相のドライを足すことになりコムが出る、クラス L/M の STFT 版）。端点は厳密分岐:
  `mix>=1` は g をそのまま使い（`1+(g−1)` の 1 ULP 丸めを避け wet パスをビット保存）、
  `mix<=0` は `1+0·(g−1)` が IEEE で厳密 1 なので分岐不要。表示（reductionDb /
  magnitudeDb）は g のまま（カーテンは検出器が削ろうとしている量で Mix 非依存）。
  これにより Mix の変化は depth 等と同じフレームレート量子化になる。
- delta 出力は **`dry − out`**（out は Mix 済みの OLA 出力）。out が Mix スケール
  済みなので delta の Mix スケールは自動的に正しく、normal `out` と delta `dry − out`
  の和は実数演算で dry に厳密一致する（IEEE では外側の減算/加算が一度丸む、2Sum
  非恒等、最大 1 ULP 残差）。
- gEff は mix についてアフィン、STFT/OLA は線形なので、実数演算で
  `delta(m) = m·delta(1) + (1−m)·delta(0)` が厳密成立する（delta(1)=dry−full_wet、
  delta(0)=dry−dry_recon=OLA 再構成残差）。**ビット恒等は成立しない**（`(1−m)·delta(0)`
  の再構成残差項が乗るため）ので、旧「delta(mix)==fl(mix·delta(1)) トレランス0」
  ゲートはアフィン恒等＋残差ゲートに置換した（同じ「Mix 非連動 delta」バグを捕捉）。
- **テスト（必須・全6レート）**: 双子（normal / delta、同一 mix・同一入力）で
  mix ∈ {0, 0.3, 0.8, 1} の各設定、全 n について:
  (i) `|normal+delta − dryAligned| <= 1e-12 · max(1, |dryAligned|)`（dryAligned は
  テスト側の自前 N 遅延。**相対 1e-12 はスペック値**。実測最大 ~1.1e−16）;
  (ii) アフィン恒等 `|delta(m) − (m·delta(1)+(1−m)·delta(0))| <= 1e-12 ·
  max(1, |dryAligned|)`（FFT 丸め蓄積 ~1e-15 相対を吸収するスペック値。実測最大
  ~5e−16。delta(1)/delta(0) は専用双子で実測）; (iii) 再構成残差
  `peakAbs(delta(0)) <= 1e-9`（steady-state n>=2N。delta(0) は OLA 再構成残差で
  実測 ~6.7e−16）。**(ii) はアフィン非線形な Mix 連動崩れを、(iii) は Mix を無視する
  全量 delta（delta(0) が removed 信号全体になる）を捕捉する** — 両者で
  「Mix 非連動 delta」クラスを封じる。

### P. Quality（オーバーラップ比）ごとの COLA を個別に保証する
- 複数の Quality/オーバーラップ設定（例 Fast 4x／Normal・High 8x）を持つ STFT
  エンジンは、**設定ごとに** OLA 正規化係数（`olaScale`）を用意すること。1 つの
  係数を全設定で使い回すと、その比率の設定でだけ depth=0 の再構成にゲイン誤差や
  段差が出る（COLA が破れる）。
- 係数は `prepare()`（またはそれに相当する準備フェーズ）で**事前計算**し、
  オーディオスレッドでは計算しない（RT安全ハードルール）。
- **テスト（必須）**: 各 Quality を個別に depth=0/mix=1 で再構成し、絶対誤差
  （例 1e-9）以内であることを**全設定・全レート**で検証。Quality 切替をまたいでも
  再構成に復帰することを別途確認する。

### Q. デュアルパス（帯域分割）合成は構造的に再構成保証する
- 信号を複数帯域に分割して独立した処理エンジンへ並列供給し、後で和を取る構成
  （デュアル解像度・マルチバンド等）は、(1) 分割フィルタが真の allpass（振幅
  フラット、位相のみ）であること、(2) 各帯域のレイテンシ差を埋めるプリディレイが
  **他帯域の生きたレイテンシから毎サンプル再計算**されること、の 2 点を構造で
  保証すること。どちらかが崩れると depth=0 でも帯域和が原信号の遅延コピーに戻らず
  コムフィルタ／振幅誤差が出る（class L/M の STFT 版）。
- **テスト（必須）**: 実装から独立した「ユニティツイン」（同一クラス・同一設定・
  同一入力のテスト側フィルタ）をオラクルに、depth=0/mix=1 の帯域和がそのツインの
  遅延版と絶対誤差（例 1e-9）以内で一致することを**全 Quality・全レート**で検証。

### R. 連続ブレンドパラメータは対数（相当）ドメインで混ぜる
- ステレオリンク量のような「0 = 独立処理 / 100 = 完全連動」を連続可変にする
  パラメータは、per-source のゲイン/係数を**線形（振幅）領域**で混ぜないこと。
  端点（0%/100%）で旧経路とビット一致しなくなり後方互換が壊れ、中間値も知覚上
  不自然になる。
- ブレンドは対数ドメイン（dB 相当）で行う: `g' = exp((1−λ)·ln(gA) + λ·ln(gB))`。
  各ソースは**独立したバリスティクス状態**を保ったままブレンドし、λ=0/1 端点は
  旧・別経路とビット一致させる。
- **テスト（必須）**: λ=0/1 端点のビット一致と、中間 λ での対数ブレンド式一致を
  全レートで検証。

### S. Listen（ノードソロ）は除去信号そのものと厳密一致させる
- 複数の処理ノード/帯域のうち 1 つだけをソロして「それが何を除去しているか」を
  聴かせる機能は、他ノードの寄与を一切混ぜないこと、かつ処理量ゼロ（depth=0 等）
  のときは除去信号がゼロ（＝再構成残差のみ）へ厳密に収束することの両方を保証する
  こと。後者を怠ると、プロファイル/ソロ機構自体が無音から幻の差分信号を作り出し
  得る（class J の亜種）。
- 実装は「既存の非ゼロ reduction を局所的にスケールするだけ」で無音から
  suppression を生成できない設計にする（ゲイン計算を depth>0 等の絶対条件でゲート
  する）。
- **テスト（必須）**: (a) 処理量>0 でソロ対象の帯域に実際の除去エネルギーがある
  こと、(b) 同一のソロ構成のまま処理量を 0 にすると出力が STFT 再構成残差程度
  （例 <=1e-6）まで収束すること、の両方を全レートで検証。

### T. Quality 切替はレイテンシ整合（PDC）を崩さない
- 解析窓長（＝申告レイテンシ）を実行時に切り替えられる STFT/FIR 処理は、切替
  直後に OLA/オーバーラップバッファが空になる瞬間を単純な wet 出力で埋めないこと
  （無音／破綻フレームが漏れる）。ホスト PDC の再申告が遅れても他トラックと
  タイミングがずれる。
- 実装: 切替が確定するフレーム境界でバッファ/ゲイン/フレーム位相をクリアしつつ、
  直後の N サンプル（N=新しい窓長）は**強制的にレイテンシ整合済みドライ**を保持
  （リフィルホールド）し、その後は既存のクロスフェード機構で wet に復帰する。
  申告レイテンシは切替直後に新しい N へ即時更新する（host PDC 再ネゴシエーション、
  オーディオスレッドから呼べる API を使う）。
- **テスト（必須）**: 全設定の遅延テーブルと、mid-stream 切替をまたいでも depth=0
  等の再構成が既定の許容内に復帰することを全レートで検証。

### U. 表示専用の平滑化は音声透過（アナライザーは音を変えない）
- アナライザー等の**表示だけ**を滑らかにする時間平滑化（`ResonanceSuppressor::
  setDisplaySmoothingMs`）は、検出／抑制／出力オーディオに一切影響してはならない。
  表示用マグニチュード（`dispMag`/`dispMagPre`）へのインプレース一次フィルタに限定し、
  `spec[]`・`gEff`・バリスティクスなど音声再構成の読み取り経路には触れない。既定 0
  （オフ）は従来の生値書き込みと bit-identical（他コンシューマ／既存テスト不変）。
- 背景: v2 のデュアル解像度で 3kHz 超の解析窓が短くなり（42.7ms→10.7ms）アナライザーが
  「速すぎ／鋭すぎ」に見えた。表示だけ ~50ms 平滑化して v1 の見た目に戻す（音は不変）。
  係数はフレームレート `fs/H` 基準で導出し hop 非依存（低／高エンジンとも同じ wall-clock τ）。
- **テスト（必須）**: `setDisplaySmoothingMs(0)` と `(50)` で同一入力を通し、**出力
  オーディオが全レートでビット一致**することを検証（音声透過）。併せて平滑化が実際に
  効いている（オンセット直後は表示が遅れて立ち上がる）ことと定常で両者が収束することも
  確認し、無効化・過剰平滑の両方向の回帰を塞ぐ。

### V. 帰還ノードは denormal storm を起こさない（直列時の致命的高負荷）
- フィードバックを持つ IIR（`Biquad` と、それを composす `LinkwitzRiley` /
  `Crossover3` / `Crossover5` / `MultiResSuppressor`）は、減衰テールが**無音（厳密
  ゼロ入力）**に沈むとき、帰還状態 z1/z2 が subnormal（非正規化）域に入り込んで
  **ピン留め**され得る。subnormal を含む浮動小数演算はマイクロコード化され、正規演算の
  **~80x** に達する（実測: LinkwitzRiley が無音入力で 398 ns/samp、フラッシュ後 5-8
  ns/samp）。1 インスタンスでは「重い」で済んでも、**2 段直列**では段 1 の減衰テールが
  段 2 の入力に underflow し、両段が同時にストームして RT 予算を突破する（実 DAW で
  resonance-suppressor 2 直列が「単に重い」ではなく致命的に。**shipping JUCE v2.1.0
  でも再現** = 共有 DSP のバグで CLAP/Visage グルーではない）。
- DSP コアは **FP モード非依存**（CPU の丸めモードを仮定も設定もしない — `shell/
  include/factory_shell/DenormalGuard.h` のコメント参照）。ホスト境界の FTZ/DAZ ガード
  （wrapper 側の `ScopedNoDenormals`）に**依存しない**こと。ガードは正しくスコープされて
  いても、コアがそもそも denormal を撒くこと自体が潜在バグ。したがって修正は core の
  帰還プリミティブ自身に置く: `factory_core::Biquad::processSample` が状態 z1/z2 を毎
  サンプル `flushDenormalToZero`（`|x| < std::numeric_limits<double>::min()` のとき
  だけ 0 に）。**既に subnormal な値だけ**を触るので正規域の出力はビット不変（既存
  オラクル・RsCore の byte-equivalence ゲートは不変。両 RS 経路が同一 `Biquad` を composす
  ので equivalence は構造的に保たれる）。純粋算術なので FTZ の有無に関わらず効く。
- **テスト（必須・全6レート）**: ハードウェアの denormal フラッシュを**明示的に OFF**
  （FTZ/DAZ クリア）にして、純粋算術ガード単体を検証する（FTZ が ON だと CPU が
  subnormal をゼロ化してしまい、ガードを外しても緑になる = **偽陽性**）。減衰バースト→
  長い無音で、(a) `allFinite`、(b) settled 域（テール後半）に**持続 subnormal なし**
  （`noSubnormals`）、(c) 出力が**厳密ゼロへ収束**（末尾 10ms が bit-zero）を assert。
  減衰途中に 1 サンプルだけ出る過渡 subnormal 出力は無害（非持続・レート依存）なので
  **settled 域**で判定する。ゲート: `core/tests/primitives_test.cpp` の
  `testDenormalFlush`（生 `Biquad`＋`LinkwitzRiley`）と、
  `plugins/resonance-suppressor/tests/dsp_test.cpp` の `chainedSeriesDenormalTest`
  （`MultiResSuppressor` を**2 段直列**、music→無音で両段が厳密ゼロへ収束）。
- **core 変更の波及**: `Biquad` は多数のプラグイン（dynamic-eq / fuzznari /
  multiband-enhancer / vocal-mbcomp / nam-player / resonance-suppressor）が composす
  ので、変更時は**全依存プラグインの DSP テスト**を走らせること（`CLAUDE.md`）。

### W. 毎ブロック冗長計算をキャッシュする（生 CPU 過負荷・直列で RT 予算突破）
- クラス V（denormal storm）の**別種**。こちらは denormal でも content 依存でも
  ない**純粋な生 CPU**。`processBlock` が、パラメータもエディタ状態も変わらないのに
  重い純関数を**毎ブロック**再計算していると、ブロックが小さい／N が大きい設定
  （例 192kHz）で固定オーバーヘッドが償却されず、2 直列で RT 予算を突破する。
  診断の分岐点: **FTZ ON でも再現する**なら denormal ではない。stage2 が stage1 の
  抑圧後（ノッチ）スペクトルを処理しても**超線形化しない**なら content 依存でもない
  （`ResonanceSuppressor::computeGains` はピーク探索ループを持たず全ビン O(N/2) 固定
  ＝検出コストは入力内容に依らない）。残るのは生コスト＝償却の問題。
- resonance-suppressor の 2 大ホットスポット（いずれも毎ブロック・内容非依存）:
  1. **プロファイルのラスタライズ**（`rasterizeProfile`）: 全ビン × 全ノードで
     `reductionProfileLinearAt`（1 ビンあたり ~20 個の transcendental、8 バンドで最重）
     を両エンジンのグリッドに掃引。実測 48k/128/Normal でエンジン本体（4 FFT）の ~2 倍。
  2. **表示スペクトルの publish**: `magnitudeDb`/`reductionDb`/`magnitudePreDb`（各
     O(N) の log10＋log-f マージ）＋ per-bin atomic store。**エディタを閉じていても走る**。
- 修正（**低リスクの純キャッシュ＋ゲート**。sonic/taste でも tolerance 緩和でもない）:
  - **(A) プロファイルキャッシュ**: ラスタライズは純関数 = f(listen ノード, reduction
    ノード, 両グリッドのビン数)。この**キー**を前ブロックと比較し、変わったブロック
    だけ再ラスタライズ（`updateProfile`）。engine は前回のプロファイルを保持
    （`setProfile` がコピー）。キーは「実際のラスタライズ入力そのもの」を比較するので
    **set フック依存より安全**（取りこぼしで stale にならない）。Quality 切替は
    グリッドのビン数変化として、listen 変化は listen キーとして捕捉。band の中心
    log 周波数（bin 掃引で loop-invariant）は `prepareBandLogF` で掃引外へ hoist
    （`reductionProfile*AtPrepared`、ビット不変）。
  - **(B) エディタ未アタッチ時は表示を止める**: publish と表示平滑化は GUI 専用。
    `editorActive`（JUCE は editor ctor/dtor、CLAP は `RsClapEditor::create/destroy`
    → `RsFeed::setDisplayActive` → `RsCore`）が false のとき両方を skip。**DSP・出力
    音声は不変**（表示ゲートは音声透過）。既定 false（GUI の無いホストは表示に払わない）。
  - JUCE 処理（`PluginProcessor`）と `RsCore` で**バイト等価な同一ロジック**にする
    （両者が共有 `factory_core::reductionNodesIdentical` とグリッドキーで同期無効化）。
- **RT 安全**: 無効化パスは比較のみ（alloc/lock 無し）。edit が来たブロックの
  再ラスタライズは**従来の毎ブロックと同じ**ワークが**稀に**走るだけ（最悪単一ブロック
  コストは従来以下。hoist でむしろ軽い）。
- **テスト（必須）**: キャッシュ出力が「毎ブロック再ラスタライズ」と厳密一致すること、
  かつ JUCE/RsCore が乖離しないことを、次の 4 ゲートで担保する:
  1. `profilePreparedHoistIdentityTest`（`dsp_test`）: `reductionProfile*AtPrepared`
     ==`*At` を 8 バンド＋両カット構成・20Hz–20kHz で**ビット恒等（tol 0）**。
  2. `reductionNodesIdentityTest`（`dsp_test`）: `reductionNodesIdentical` が全カット
     ＋全バンドの**各フィールド**の差分を検出する（＝キャッシュキーの取りこぼし無し）。
     これが無いと「あるフィールドを見落とす比較バグ」を**両側が同じく持てば
     byte-equivalence では捕捉できない**（両方が同じく stale）。
  3. `rscore_equiv` の **automation ケース**（`runAutomation`）: **毎ブロック**
     パラメータを変える（band/cut 各フィールド・depth・時折 Quality/Listen）ことで
     キャッシュを常時無効化し、出力＋published スペクトルが**バイト一致**することを
     検証（両側のキャッシュが同期して無効化される保証）。静的 config の equivalence
     だけでは非対称／フィールド盲目な無効化を捕捉できない。
  4. `rscore_equiv` の **editor-closed ケース**: `editorActive=false` で音声が
     **バイト不変**（表示ゲートが音声透過）かつ両側が published スペクトルを同一に
     凍結することを検証。
- **診断メモ（再発時の切り分け）**: 生コスト過負荷は「CPU% が **1/ブロックサイズ**
  と **有効ノード数**に比例、**入力内容には非依存（無音でも同じ）**」が指紋。
  block を 512→128 に落として CPU% がほぼ倍化するなら固定オーバーヘッドの償却不足
  （＝このクラス）。denormal（V）は FTZ ON で消え、content 依存は stage 直列で超線形化
  する — どちらでもなければ W。

## チェックリスト

### 新規プラグインを追加するとき
- [ ] DSP は `juce::AudioProcessor` から分離した core クラスで、headless に
      テスト可能（GUI/ホスト不要）。
- [ ] テストは `sampleRatesFromArgs()` を使い **全6レート** で走る。CMake の
      `foreach(_fs ...)` も同じ6レート。
- [ ] フィードバックがあるなら A（loop gain<1・最悪ケース）・C（finite ガード）・
      V（減衰テールが無音で denormal を持続させない）を検証。
- [ ] 共振/高次フィルタがあるなら B（ピーク絶対上限）を検証。
- [ ] 遅延バッファがあるなら D（最悪ケースで確保）を検証。
- [ ] `prepareToPlay` で `reset()`、bypass/mode 遷移で状態リセット（E）。
- [ ] 連続パラメータは平滑化（F）、状態は prepareToPlay で確保（RT安全）。
- [ ] アナライザー/STFT は `fftOrderForSampleRate`＋解像度不変量（G）。
- [ ] 量的検証は独立静的オラクル（H）。バグ経路を実際に踏む。
- [ ] GUI/audio 共有スカラは atomic（I）。検出器には絶対 floor（J）。
- [ ] 表示桁・カーブ整合・ラベル幅は共有 UI ヘルパで（K）。
- [ ] ドライ×ウェットを並列合成するなら L（位相コヒーレント・コムフィルタ回避）を検証。

### `core/` を変更するとき
- [ ] **全依存プラグインのテスト**を走らせる（`CLAUDE.md`）。
- [ ] 挙動・レイテンシ・tolerance・oracle を変える場合は **人間承認**。
- [ ] 追加した不変量は `DspInvariants.h` に共通化できないか検討する。
